Nezha: Deployable and High-Performance Consensus Using Synchronized Clocks

十二月 19, 2024 · map[name:atditto]

Nezha: 可部署且高性能的共识协议#

大家好，今天我要给大家介绍一篇来自 VLDB 2023 的论文，题目是《Nezha: Deployable and High-Performance Consensus Using Synchronized Clocks》。这篇论文由斯坦福大学和纽约大学的研究团队共同完成，作者包括 Jinkun Geng、Anirudh Sivaraman、Balaji Prabhakar 和 Mendel Rosenblum。

背景介绍#

在现代数据库和微秒级应用中，高性能的共识协议至关重要。这些应用需要高吞吐量和低延迟的共识协议来确保数据一致性和容错性。然而，现有的共识协议如 Multi-Paxos 和 Raft 虽然可以广泛部署，但其性能存在局限性。而像 NOPaxos 和 Speculative Paxos 这样的高性能替代方案，则需要云服务提供商的技术支持，普通云租户无法轻易获得。

基于这个背景，Nezha 的目标是开发一种高性能的共识协议，能够在公共云环境中由租户自行部署，而不依赖于云服务提供商的支持。

Nezha 的核心思想#

Nezha 的核心思想包括以下几个方面：

  1. Deadline-Ordered Multicast (DOM)：通过高精度时钟同步减少公共云中的消息重排。
  2. 推测执行（Speculative Execution）：在请求提交之前执行请求，以减少延迟。
  3. 代理部署（Proxy Deployment）：使用无状态代理来减轻客户端的计算负担。
  4. 每个接收者接收到的消息序列的最长递增子序列（Longest Increasing Subsequence, LIS）。

公共云环境中的重排问题#

文章的动机是公共云环境中的重排问题。具体来说，在公共云环境中，由于网络路径的不同，消息到达不同接收者的顺序可能不同，导致重排现象频繁发生，进而增加了维护分布式系统一致性的开销。

作者通过在 Google Cloud 上的实验发现，在公共云环境中，消息重排现象非常普遍。这三个图表共同展示了在 Google Cloud 上，重排序得分（Reordering Score）是如何受到提交速率、发送者数量以及使用不同百分位估计的 DOM 的影响的。

重排序得分越高，表示公有云中发生的重排序越多。图 1 显示，当我们改变提交率，保持发送者的数量固定为 2 时，重新排序的得分很快超过 28%。在图 2 中，当我们改变发送者的数量时，保持提交率固定在 10K 条消息/秒，重新排序得分随着发送者数量的增加而迅速增加到 43%。

这导致乐观协议（如 Fast Paxos）在公共云环境中性能下降，因为它们依赖于消息的一致到达顺序。

DOM 的原理#

DOM 的原理是：

  • 发送者：为消息附加全局时间戳的截止时间，并将其多播给所有接收者。
  • 接收者：在截止时间或之后处理消息，并按截止时间顺序处理多个消息。

DOM 的特点是“尽力而为”，它不保证消息的可靠传输，只确保消息按截止时间顺序处理。截止时间的设置基于最近的一向延迟（OWD）估计，以平衡避免重排和增加等待时间。

单向延迟（One-Way Delay, OWD）#

单向延迟（One-Way Delay, OWD）是指从发送者发送消息到接收者接收到消息的时间间隔。

测量方法：

  • 时钟同步：通过高精度时钟同步技术，发送者和接收者的时钟被同步到一个共享的全局时间。
  • 时间戳：发送者在发送消息时附上发送时间戳，接收者在接收到消息时记录接收时间戳。
  • 计算 OWD：接收者通过减去发送时间戳和接收时间戳来计算 OWD。

DOM 的截止时间设置：DOM 根据最近的 OWD 估计值设置消息的截止时间，以平衡避免重排和增加等待时间的权衡。

Nezha 的模型和假设#

Nezha 的模型和假设包括：

  • 故障停止模型：假设副本可能因进程崩溃或服务器重启而失败，但不处理拜占庭故障。
  • 2f+1 实例：包括 1 个领导者和 2f 个追随者，最多允许 f 个实例同时崩溃。
  • 副本的状态：包括 replica-id、view-id、status、early-buffer、late-buffer、Log、sync-point、commit-point 等。

无状态代理（Stateless Proxy）#

Nezha 中的无状态代理（Stateless Proxy）是一个关键组件，旨在减轻客户端的计算负担，并提高系统的可扩展性和容错性。它的功能有：

  • 代理负责将客户端的请求多播给所有副本。
  • 代理负责收集副本的回复并进行法定人数检查，以确认请求已提交。
  • 代理负责与副本进行时钟同步，确保所有副本和代理使用相同的全局时间。

快速路径和慢速路径#

Nezha 的另一个核心组件是快速路径和慢速路径。

快速路径关键点#

  • 超法定人数（super quorum）：快速路径需要领导者和 f+⌈f/2⌉ 个跟随者的快速回复，而不是简单的法定人数（f+1）。这是因为快速路径中没有领导者与跟随者之间的通信，简单的法定人数无法保证日志的一致性。
  • 投机执行：领导者在请求提交之前执行请求，并将执行结果发送给客户端。客户端只有在收到法定人数的回复后才接受执行结果，否则客户端会重试请求。

慢速路径关键点#

  • 日志一致性：慢速路径通过广播日志修改消息，强制跟随者保持与领导者的日志一致，确保所有副本的日志在请求提交时是一致的。
  • 执行和提交解耦：领导者在请求提交之前执行请求，并将执行结果发送给客户端。客户端只有在收到法定人数的回复后才接受执行结果，否则客户端会重试请求。这种解耦设计使得慢速路径的提交过程比其他乐观协议更快。

快速路径基本流程#

  1. 请求多播：客户端将请求发送给代理，代理将请求附上发送时间和截止时间，然后多播给所有副本。
  2. 早期缓冲区（early-buffer）：如果请求的截止时间大于早期缓冲区中最后一个释放的请求的截止时间，请求进入早期缓冲区。
  3. 日志追加：请求在截止时间到达后被释放，并追加到副本的日志中。日志按请求的截止时间排序。
  4. 执行和回复：领导者执行请求并发送包含执行结果的快速回复，跟随者发送不包含执行结果的快速回复。
  5. 法定人数检查：代理收集领导者和 f+⌈f/2⌉ 个跟随者的快速回复后，确认请求提交。

慢速路径基本流程#

  1. 请求处理：当请求因延迟或丢失而无法进入早期缓冲区时，请求进入晚期缓冲区（late-buffer）。
  2. 截止时间修改：领导者修改请求的截止时间，使其能够进入早期缓冲区。
  3. 日志追加和广播：领导者追加请求到日志，并广播日志修改消息，强制跟随者保持与领导者的日志一致。
  4. 跟随者处理：跟随者在收到日志修改消息后，检查并修改其日志，确保与领导者的日志一致。
  5. 慢速回复：跟随者在日志一致后发送慢速回复。
  6. 法定人数检查：代理收集领导者和 f 个跟随者的慢速回复后，确认请求提交。

Nezha 的其他考虑#

  • 代理故障：代理故障不会影响系统的正确性。
  • 客户端超时：客户端在等待代理回复时启动计时器。
  • 重试机制：客户端可以选择使用相同的代理或不同的代理，代理会重新提交请求，并可能使用不同的发送时间和延迟界限。
  • 最多一次语义：副本维护最多一次语义（例如 <client-id, request-id>）。

恢复机制#

Nezha 的恢复机制基于故障停止模型和稳定存储，包括：

  • 副本重新加入：失败的副本以追随者身份重新加入。
  • 领导者变更：如果失败的副本是领导者，剩余的追随者在未收到领导者心跳后发起视图变更，并选举新的领导者。
  • 新领导者的日志恢复：新领导者从其他副本获取日志、同步点和最后一个正常视图，恢复其日志并执行日志条目。

优化手段#

  • 增量哈希：通过维护运行哈希值，每次追加或修改日志条目时快速更新哈希值，避免重新计算整个日志的哈希。
  • 交换性优化：对于操作不同键的请求，允许它们以任意顺序执行，从而提高快速提交比率。
  • 定期检查点：领导者定期选择最小的 sync-point 作为 commit-point，并广播给所有副本，副本在 commit-point 处进行状态检查点，加速恢复过程。

实验结果#

作者在 Google Cloud 上进行了测试，使用 n1-standard-4 虚拟机作为客户端，n1-standard-16 虚拟机作为副本和 NOPaxos 序列器，n1-standard-32 虚拟机作为 Nezha 代理。与 Multi-Paxos、Fast Paxos、NOPaxos、Raft、Domino 和 TOQ-EPaxos 等协议进行了对比。

评估方法#

  • 闭环测试：客户端在收到上一个请求的回复后才发送下一个请求。
  • 开环测试：客户端根据泊松过程发送请求，无需等待上一个请求的回复。

Nezha 与 TOQ-EPaxos 比较#

  • 同步机制：

    • TOQ-EPaxos：仅同步副本以减少副本之间的消息重排序，这有助于副本以相同顺序处理非交换请求。
    • Nezha：同步副本和代理，以减少从代理到副本的重新排序。Nezha 在共识工作流中控制了更多的路径，使得在客户端和副本位于不同区域时，Nezha 仍能有效减少延迟。

  • 顺序一致性：

    • TOQ-EPaxos：不能保证顺序的一致性，不同副本可能以不同顺序处理请求。
    • Nezha（DOM）：优先考虑一致排序，确保任何两个副本不会以不同顺序释放相同请求。DOM 的设计基于 Nezha 的单 leader 设计，可以快速修复集合不等式。

  • 性能表现：

    • TOQ-EPaxos：在局域网中的性能不如 Multi-Paxos，即使在相同框架下实现。
    • Nezha：在局域网和广域网设置中都能产生良好的性能，表现出通常高性能的协议特性。

Nezha 与 Multi-Paxos、Fast Paxos 和 NOPaxos 的比较#

  • Fast Paxos：在开环测试中表现最差，由于频繁的请求重新排序，导致其进入慢路径，吞吐量显著降低。
  • NOPaxos：在开环测试中表现不佳，主要因为单线程处理请求和间隙处理导致性能瓶颈。通过优化（NOPaxos-Optim）使用多线程处理，性能有所提升，但仍受限于开环测试中的请求爆发和数据包重新排序。
  • Nezha：表现最佳，实现了最高的吞吐量和最低的延迟。其优势来自于三个设计方面：一致的排序机制（DOM）、将请求执行和仲裁检查分离、以及使用交换性减少延迟。Nezha-非代理版本由于代理与客户端共存，进一步降低了延迟。

消融实验#

每次从 Nezha 的完整协议中删除一个组件，得到三个变体 NoDOM、No-QC-Offloading、No-Commutativity。图 7 显示，删除这三个组件中的任何一个都会降低性能（即吞吐量或延迟）。

  • No-DOM 变体：

    • 移除 DOM 原语后，请求不再按照截止日期排序，导致一致排序和集合相等无法保证。
    • 这种变体实际上成为具有仲裁检查卸载的 Multi-Paxos 协议，leader 副本负责排序和请求多播，导致吞吐量降低和延迟增加。

  • No-QC-Offloading 变体：

    • 仍然使用 DOM 进行排序和请求多播，但依赖 leader 进行每个请求的 quorum 检查。
    • leader 的负担加重，导致 leader 副本成为瓶颈，降低了吞吐量和延迟性能。

  • No-Commutativity 变体：

    • 禁用交换性优化，降低了快速提交比率，导致更多请求通过慢路径提交。
    • 对吞吐量影响不明显，但延迟性能降低了 24.2%。

FCR 和 FPL#

  • FCR = （快速路径提交的请求数）/（总请求数）
  • FPL（Fast Path Latency）：是指请求在快速路径中提交的延迟。
  • OCL（Overall Commit Latency）：是指从客户端提交请求到系统确认请求已提交并返回执行结果的总体延迟，包括了请求在系统中的所有处理时间。

百分位数的影响#

  • FPL 和 FCR 的权衡：百分位数越大，FCR 越高（即更多请求通过快速路径提交），但 FPL 越长（即快速路径上的延迟增加）。
  • 总体提交延迟（OCL）：FCR 和 FPL 都会影响所有请求的总体提交延迟。

交换性优化的影响#

  • 无交换性优化（图 8a）：

    • 使用更大的百分位数（从 50p 到 75p）时，FCR 的改善超过了 FPL 的增加，导致 OCL 降低。
    • 当继续使用更大的百分位数（如 95p 和 99p）时，尽管 FCR 不断增长，但由于早期缓冲中保持延迟的增加，FPL 变得更长，破坏了快速路径的好处，不再有助于减少 OCL。

  • 有交换性优化（图 8b）：

    • 使用 50p 已经达到了很高的 FCR，因此使用较大的百分位数几乎不能改善 FCR，而只能增加 FPL，进而增加 OCL。
    • 因此，选择 50p 作为 DOM 中的百分位数，并验证了其在不同工作负载下的鲁棒性。

Nezha-代理（Nezha-Proxy）#

  • 使用 5 个代理，每个请求发送 9 条消息并接收 17 条消息（9 个快速回复和 8 个慢速回复）。
  • 相对于 Nezha-非代理，Nezha-代理产生 2 个额外的消息延迟，但在客户端显著减少了 CPU 占用。
  • 随着吞吐量的增长，Nezha-代理甚至实现了比 Nezha-非代理更低的延迟，因为 Nezha-非代理使客户端 CPU 密集型。

Nezha-非代理（Nezha-Non-Proxy）#

  • 每个请求发送 9 条消息并接收 17 条消息。
  • 随着吞吐量的增长，客户端 CPU 占用显著增加，导致延迟增加。
  • 最大吞吐量比较（图 9b）：
    • 对于相同的 CPU 资源，没有代理的客户端的吞吐量随着副本数量的增加而明显下降。
    • 这种瓶颈也可能发生在具有类似卸载设计的其他协议中（例如，[24,38,40,53]）。
    • 相比之下，当配备代理时，无论副本的数量如何，客户端都保持高吞吐量。

广域网环境中的性能#

  • 实验设置：

    • 在 5 个区域运行开环测试：3 个副本分别位于欧洲-北 1-a、亚洲-东北 1-a 和南美洲-东 1-a；10 个开环客户端分为两组，分布在 us-east1-b 和 us-west1-a；相应地，两个代理也分布在 us-east1-b 和 us-west1-a 中，以服务于其区域中的客户端。

  • 性能比较：

    • 与局域网评估相比（图 6），Nezha 在广域网中的表现甚至超过了所有基线。
    • 每个客户端使用一个线程提交请求，另一个线程处理应答。

  • 具体表现：

    • NOPaxos-Optim：在四个基线中表现最好，但 Nezha 的延迟和吞吐量仍然比 NOPaxos-Optim 高出 1.51 倍和 2.55 倍。
    • TOQ-EPaxos：当客户端和副本位于不同区域时，TOQ 对减少 EPaxos 的延迟帮助不大，因为 TOQ 仅利用时钟同步来减少副本之间的冲突，快速路径仍然需要 2 WAN RTT。
    • Nezha：由于代理是无状态的，通常可部署在与客户端同一区域，使客户端-代理延迟成为局域网消息延迟。因此，Nezha 可以在快速路径上实现 1 WAN RTT。

Nezha 在基于磁盘的版本中与 Raft 的性能比较#

  • 日志持久性：

    • Raft：在日志持久性上建立其正确性，依赖于稳定的存储来实现更强的容错性（例如，电源故障）。
    • Nezha：为了与 Raft 进行公平比较，将 Nezha 从无磁盘操作转换为基于磁盘的版本，使其在副本发送回复之前，首先将相应的日志条目（包括视图 ID 和崩溃向量）持久化到稳定的存储中。如果副本重新启动，它可以恢复其状态并重播快速回复/慢速回复。

  • 性能比较：

    • Raft-1：来自 [36] 的开源实现。
    • Raft-2：基于 [23] 的 Multi-Paxos 代码作为起点实现。
    • Nezha：在闭环（图 11）和开环测试 [9] 中都优于 Raft。

  • 延迟表现：

    • 在 Nezha 中有或没有代理的延迟几乎没有区别，因为延迟现在由磁盘写控制，而不是消息延迟。

Redis#

  • 工作负载：选择 YCSB-A 作为工作负载，使用 HMSET 和 HGETALL 操作 1000 个键。
  • 客户端设置：使用 20 个闭环客户端来饱和未复制 Redis 的处理能力。

    • Nezha 的吞吐量是未复制系统的 5.9%。

  • 客户端设置：启动 48 个参与者和 16 个网关，其中 3 个参与者连接到一个网关。

总结#

Nezha 在闭环和开环测试中，无论是在吞吐量还是延迟方面，都显著优于所有基准测试。代理的使用显著减少了客户端的 CPU 开销，尤其是在副本数量较多的情况下。在广域网环境中，Nezha 的性能优势更加明显。

Nezha 是一个高性能的共识协议，能够在公共云环境中轻松部署，通过 DOM 和时钟同步技术显著提升性能。未来的工作包括将 Nezha 替换为现有系统（如 Kubernetes）中的 Multi-Paxos/Raft 后端，以及使用 DOM 改进现有的并发控制算法或发明新的并发控制协议。
